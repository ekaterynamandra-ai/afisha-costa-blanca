# /approve

Одобряет неделю для автопубликации через GitHub Actions.

## Использование

- `/approve 4` — всю неделю
- `/approve w4-fri-1` — один пост
- `/approve 4 --dry` — валидация без сохранения (проверить что всё ок)

## Что делает

1. Находит `content/schedule/weekN-*.json`
2. Для каждого поста проверяет:
   - `text` не пустой
   - `photo` указан и файл существует (или URL валидный)
   - `date`, `time`, `type` заполнены
   - HTML-теги валидны: `<b>`, `<i>`, `<a href>` закрыты; нет битых `<`/`>`
   - Дата в будущем (не просрочена)
3. Если ВСЁ ок:
   - На уровне файла ставит `"approved": true`
   - Для каждого поста `status: draft` → `status: approved`
4. Если ЕСТЬ ошибки:
   - Показывает список: "4 поста без фото, 1 битый HTML"
   - НЕ одобряет
5. Подтверждение: "Week 4 approved. 6 постов на автопубликацию Ср 22 → Вс 26 апреля."

## Важно — правило проекта

- **Никогда не публиковать без одобрения пользователя!** /approve — это именно одобрение пользователя.
- Перед /approve я показываю HTML-превью и жду явного «ок» / «одобряю» / «да»
- В сомнении — уточни конкретные посты

## Отмена одобрения

Вручную в JSON: `"approved": false`, для нужных постов `status: draft`. Или `/update <id> status draft`.

## Что дальше

После /approve GitHub Actions (`.github/workflows/autopublish.yml`) запускается раз в 2 часа по cron. Проверяет, что:
- `approved: true` на уровне файла
- `status: approved` на посте
- Время наступило (`date + time` в прошлом, в пределах часа)

Публикует → обновляет `status: published`, `published_at`, `message_id`.
