# /week

Разворачивает скелет недели в полные тексты. Читает `content/schedule/weekN-*.json`, для каждого поста где `text` пустой — генерирует тело по типу поста и правилам.

## Использование

- `/week 4` — неделя по номеру
- `/week apr-22` — по дате начала
- `/week next` — ближайшая неодобренная неделя

## Что делает

1. Находит нужный `content/schedule/weekN-*.json`
2. Для каждого поста с `text: ""`:
   - Смотрит `type` → находит шаблон в `docs/content-framework.md`
   - Ищет фактические данные в:
     - `content/events/events-april-may-2026.json` (события с датами)
     - `content/data/*.md` (сырые заметки)
     - `content/google-maps-export/` (места и ссылки)
   - Пишет полный HTML-текст (с `<b>`, `<i>`, `<a href>`) по стилю предыдущих недель (смотри week3 как эталон)
   - Находит фото в `content/photos/places/` (если есть) или записывает `photo: "NEED_PHOTO"` и выводит список того, что надо скачать
3. Ставит `status: draft` → готов к ревью
4. Запускает `python scripts/generate_preview.py weekN-*.json` → HTML в `content/drafts/`
5. Показывает сводку: сколько постов развернул, сколько фото нужно, основные акценты

## Стиль (как в week3)

- Заголовок: эмодзи + 1 ёмкая строка (жирным в тексте через `<b>`)
- Первая строка: крючок в 1-2 предложения
- Блоки с `<b>эмодзи подзаголовок</b>` — что, где, когда
- Ссылки на Google Maps для каждого места
- Время в минутах от Аликанте
- Совет / лайфхак в конце
- `🎯 Подойдёт: 💑 парам · 📸 фотографам · 👨‍👩‍👧 семьям`
- `#хештеги` в конце (3-4 шт)

## Что НЕ делает

- Не меняет `approved` — это /approve
- Не публикует — это /publish_now или автопубликация
- Не скачивает фото автоматически — даёт список для ручной проверки через `python scripts/download_photo.py`

## Последовательность работы с неделей

```
/plan → /week N → [ручная правка] → /preview → /approve N → [GitHub Actions публикует]
```
