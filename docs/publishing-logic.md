# Логика публикаций Афиша Costa Blanca

> Этот файл описывает как работает автопубликация. Если что-то не так — сначала проверь здесь.

---

## Общая схема

```
cron-job.org (каждые 2 ч)
    ↓ HTTP POST с токеном
GitHub API → запускает workflow_dispatch
    ↓
GitHub Actions: autopublish.yml
    ↓
python scripts/autopublish.py
    ↓
читает content/schedule/week*.json
    ↓
для каждого approved поста:
    - проверяет дату/время
    - если время пришло → публикует в @afishaCB
    - сохраняет статус → published
    - отправляет отчёт админу в личку
```

---

## Файлы расписания

**Расположение**: `content/schedule/week*.json`

**Структура файла**:
```json
{
  "week": "2026-04-07 to 2026-04-13",   // ⚠️ должно быть Пн–Вс
  "approved": true,                       // false = ничего не публикуется из этого файла
  "posts": [
    {
      "id": 12,
      "date": "2026-04-07",               // YYYY-MM-DD
      "time": "10:00",                    // HH:MM по испанскому времени
      "title": "🏖️ Дюны Гуардамара",
      "status": "approved",               // approved | published | draft | rejected
      "photo": "https://...",             // одна фотка
      "photos": ["url1", "file2"],        // ИЛИ альбом из нескольких
      "text": "🏖️ <b>...</b>",           // HTML-форматирование
      "test_mode": false                  // true = отправить только админу
    }
  ]
}
```

---

## Статусы постов

| Статус | Значение | Что делает скрипт |
|--------|----------|-------------------|
| `draft` | Черновик, ещё не готов | Игнорирует |
| `approved` | Готов, ждёт времени | Публикует когда наступит дата+время |
| `published` | Опубликован | Игнорирует (никогда не повторяет) |
| `rejected` | Отклонён | Игнорирует навсегда |

**Также важно**: на уровне файла `approved: false` → ВСЕ посты файла игнорируются. Это позволяет готовить план целиком и потом одобрять одним движением.

## Логика выбора что публиковать

```python
для каждого файла week*.json:
    если approved == false → пропустить весь файл
    
    для каждого поста:
        если status == published → пропустить (уже)
        если status != approved → пропустить (черновик/отклонён)
        
        если post_datetime > сейчас → пропустить (рано)
        
        # Публикуем
        если есть photos[] (>1) → отправить альбом
        иначе если есть photo → отправить с фото
        иначе → отправить только текст
        
        если test_mode == true → отправить админу в личку (НЕ в канал)
        
        если успешно:
            status → published
            published_at → текущее время
            сохранить файл (СРАЗУ, чтобы не дублировать)
            отправить отчёт админу
        иначе:
            отправить ошибку админу
```

## Напоминания о планировании

### Когда приходят
- **Пятница 4** → "пора готовить план на следующую неделю"
- **Суббота 5** → "к завтра нужен план (Пн–Вс)"
- **Воскресенье 6** → если нет плана → "СРОЧНО! завтра новая неделя"
- **С 25 числа месяца** → напоминание про месячный план

### НЕ приходят
- В Пн-Чт (только если что-то опубликовалось — тогда отчёт + напоминание)
- Если план уже одобрен на следующую неделю → молчим (или коротко "✅ План одобрен до X")

### Логика проверки
```python
сегодня = текущая дата
дни_до_воскресенья = (6 - weekday) % 7
ближайшее_вс = сегодня + дни_до_воскресенья
конец_след_недели = ближайшее_вс + 7 дней

план_покрывает_след_неделю = last_approved >= конец_след_недели

если план_покрывает_след_неделю:
    "✅ План одобрен до X"
елси есть_неутверждённый_черновик:
    "📋 Есть черновик — нужно согласовать"
иначе:
    "⚠️ Нет плана"
```

## Расписание cron

**cron-job.org** делает HTTP POST каждые 2 часа:
- 8:00 испанского
- 10:00
- 12:00
- 14:00
- 16:00
- 18:00
- 20:00

**Зачем cron-job.org**: GitHub Actions scheduled cron ненадёжен — может задержаться на 30-60 мин или пропустить запуск. Внешний cron гарантирует точное время.

**Как работает**:
- cron-job.org → POST на `api.github.com/.../actions/workflows/autopublish.yml/dispatches`
- GitHub запускает workflow → запускает скрипт → публикует все "созревшие" посты

**Точность публикации**: до 2 часов задержки от запланированного времени (потому что cron каждые 2 часа). Если поставлен пост на 10:00 → опубликуется в 10:00 (cron на 10:00 запустит скрипт).

python scripts/autopublish.py

## ⚠️ Известные проблемы и решения

### Локальный диск и репо рассинхронизированы
**Симптом**: на твоём диске пост `approved`, на GitHub `published` (или наоборот)
**Причина**: GitHub Actions сохранил статус, но локально не подтянули
**Решение**: `git pull` перед каждой работой

### Пост опубликовался дважды
**Причина**: скрипт упал между публикацией и сохранением статуса
**Защита**: статус сохраняется СРАЗУ после публикации (см. `process_schedule`, line ~363)

### GitHub Actions cron не сработал
**Причина**: GitHub Actions scheduled — ненадёжный
**Решение**: используем cron-job.org как внешний триггер

### Дубликат напоминания "план одобрен"
**Причина**: каждый запуск cron вызывал check_planning_reminders
**Решение** (текущее): напоминания только в пт/сб/вс ИЛИ если что-то опубликовалось
**Возможное улучшение**: маркер `last_reminder_date.txt` чтобы отправлять не чаще 1 раз/день

## Что можно улучшить (будущее)

### 1. Дедупликация напоминаний
Сейчас если cron сработает 4 раза в день в воскресенье — придёт 4 одинаковых напоминания. Решение: файл `.last-reminder` с датой последней отправки.

### 2. Превью перед публикацией
За 1 час до публикации отправлять админу превью с кнопкой "✅ Опубликовать / ❌ Отменить".

### 3. Автоматическая генерация альбомов
Если в посте `photos: ["folder/*.jpg"]` — взять все файлы из папки.

### 4. Аналитика после публикации
Через час после публикации подтянуть статистику Telegram (просмотры, реакции) и сохранить в посте.

### 5. Архивация прошедших недель
Старые `week*.json` где все посты `published` → переносить в `content/archive/`.

### 6. Валидация перед публикацией
- Проверка что фото URL живой
- Проверка длины caption (Telegram limit 1024)
- Проверка HTML-валидности

### 7. Мульти-канальность
Поддержка нескольких каналов из одного поста (например, дублировать в Instagram).
